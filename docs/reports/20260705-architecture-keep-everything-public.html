ScratchPad · 架构审查报告

“全员 public” 不是命名问题，
是缺失的接缝被 TS 迁移照了出来

TypeScript 迁移里那句“只能把所有成员保持 public”不是迁移的副作用——它是把早就存在的跨模块耦合翻译成了编译器能看见的形状。每一个被迫 public 的下划线成员，背后都是一个本该存在、却没有的接缝（seam）。这份报告把那些泄漏映射成 4 个深化（deepening）候选。

生成：2026-07-05 · 关注面：Board / InputController / TextManager / export 的跨模块内部访问 · 无 CONTEXT.md / ADR（本仓库尚未建立） · 本文件为提议，非实现
核心论点。 用删除测试（deletion test）看每个泄漏：如果把泄漏点“封起来”，复杂度会消失，还是会在多个调用方之间重新冒出来？board.strokes 被 input / textbox / export 三方直接 push / filter / 赋值——把它设成 private，复杂度会立刻在三个模块里炸开（bbox 不变量、live↔commit 转换、文字块对账）。这说明缺的不是 private 关键字，而是一个拥有“笔画文档”这件事的深模块。其余三个泄漏同理，各对应一条没被切出来的接缝。

候选 A · 把“笔画文档”从 Board 神对象里切出来

Candidate A

StrokeStore — 拥有 strokes 数组 + 所有变更 + bbox 不变量的深模块

Strong
Files
src/board.tssrc/input.tssrc/textbox.tssrc/export.ts（新增 src/stroke-store.ts）
Problem

Board 是个神对象：它同时是笔画仓库、canvas 渲染器、选区/套索视觉状态和viewport。因为它把 strokes 数组裸露出来，调用方绕过它的方法直接改数组——input.ts 里 board.strokes.push(s)（撤销重插）、board.strokes = …filter(x!==lt.stroke)（双击撤点）；textbox.ts 里 board.strokes.push(stroke) / = …filter()。于是“每条 stroke 必须有合法 bbox”“live 是 number[]、commit 后是 Float32Array”“文字块 vs 手写”的不变量散落在三个模块里。而且这堆纯几何逻辑（hitTest / 多边形选区 / bbox）挂在 Board 上，要造一个真 <canvas> 才能实例化（构造函数 getContext + resize + addEventListener），所以现在无法 headless 单测。

Solution

把“笔画文档”这件事切进 StrokeStore：拥有私有的 strokes，对外只给一个小接口——add(stroke) / remove(ids) / replace(strokes) / translate(strokes,dx,dy) / hitTest(x,y,r) / inPolygon(poly) / forEach(cb) + 一个 change 事件；bbox 不变量在内部维护。Board 退化成“在 StrokeStore 上渲染一个 viewport + 管选区/套索的视觉状态”。input / textbox 改成调 store 方法，不再碰数组。strokes 随之变 private。

Benefits
  • Locality：“改一条 stroke 会怎样”只在 StrokeStore 里。bbox bug、选区几何 bug 全部收敛到一个文件。
  • Leverage：input/textbox 拿到“加/删/移一批 stroke，bbox 自动对”的高杠杆接口，不再各自维护数组。
  • Tests：接口即测试面。hitTest、多边形选区、bbox 上卷、撤销驱动的增删——全都能不碰 DOM 单测（今天做不到）。
  • 直接消灭泄漏：board.strokes 变 private，6 处裸数组变更消失。
Deletion test · 删掉 StrokeStore → 笔画变更逻辑 + bbox 不变量重新炸回 board/input/textbox 三处。复杂度集中，不是被搬运 → 它挣得起自己的位置。
⚠️ 与家族风险分区一致：这是内存态 app 层文档模型，跟 db.ts（IndexedDB 持久化）和家族的云同步 store 库无关——ScratchPad 本就纯本地无云。StrokeStore 不是那条红线。

候选 B · 用“渲染后”接缝替换 board.render 猴补丁

Candidate B

渲染观察者 — Board.addRenderListener(fn)，取代运行时改写方法

Strong
Files
src/app.tssrc/board.ts
Problem

app.ts:337 直接 board.render = function(){ origRender(); textManager.updateOverlayTransform(); syncOverlay(); updateZoomLabel(); positionSelChip(); }——运行时把一个方法整个换掉，只为在每帧后挂 4 件事。这是红旗：调用方为了插行为而改写被调方的实现；也让 render 永远不能变 private，且顺序耦合藏在补丁里。

Solution

Board 暴露一个渲染后接缝：addRenderListener(fn)，在 render() 末尾依次调用已注册的监听器。app 注册它的“浮层同步 + HUD 更新”，不再重赋值方法。render 可收回成内部方法。

BEFORE · 猴补丁
appboard.render = fn
覆盖 Board.render
origRender() + overlay + HUD + chip
顺序 & 依赖藏在补丁里；render 不可 private。
→
AFTER · 监听器接缝
Board.render()末尾 fire listeners
app→ addRenderListener(overlay+HUD)
未来任意订阅者可加，互不干扰
render 变 private；一个真接缝，两个潜在适配器。
Benefits
  • Locality：“每帧后要干什么”从一段被覆盖的匿名函数，变成显式注册列表。
  • Leverage：再有订阅者（如 minimap、性能计数）时直接 addRenderListener，不用叠补丁。
  • 消灭泄漏：render 收回 private，去掉运行时方法改写这个红旗。
Deletion test · 接缝较浅（单一订阅者=假想接缝），但它消灭的是一个确定的红旗（运行时改写方法）。价值在“让 render 可封装”，而非杠杆。

候选 C · 让撤销历史成为两个编辑器共享的属主

Candidate C

History 模块 — 公共 push(entry)，替换跨模块的 input._pushUndo

Worth exploring
Files
src/input.tssrc/app.tssrc/textbox.ts（新增 src/history.ts）
Problem

撤销/重做栈住在 InputController 里，但文字块的增删（TextManager → app 的 onAdd/onDelete 回调）从外部调 input._pushUndo({type:"add",…}) 把历史条目塞进去。下划线本意是“别从外面叫”，可它偏偏被外面叫。历史其实是手写编辑和文字编辑共享的关注点，却被 InputController 顺手拿着，没有干净属主。UndoEntry 类型已在 types.ts，说明数据早就是共享的。

Solution

把 undo/redo 两个栈 + push / undo / redo / clear 提进 History 模块，push(entry) 是正当的公共接口。InputController、TextManager、选区移动都经 history.push()。input._pushUndo 的跨模块下划线调用消失。（条目的 apply/revert——撤销“add”=删 stroke、撤销“erase”=按原 id 重插——可留在一个 applier 里，由 History 驱动。）

BEFORE · 顺手拿着
InputController
undoStack / redoStack
app / textbox→ input._pushUndo →
文字编辑从外面戳 input 的下划线方法。
→
AFTER · 共享属主
Historypush · undo · redo
input→ history.push →
textbox→ history.push →
一个公共接口，两个真实适配器 → 真接缝。
Benefits
  • Locality：“撤销栈怎么长、封顶 100、新动作清 redo”只在 History 里。
  • Tests：History 的 push/undo/redo 状态机可脱离 pointer 事件单测。
  • 消灭泄漏：_pushUndo 从“下划线跨模块”变成 history.push 公共调用。
Deletion test · 中等。History 集中了撤销状态机；但 apply/revert 仍需 store+db 依赖，切得不够干净时收益会打折。先做 A 之后，apply 逻辑能干净地落在 StrokeStore 上，C 会顺很多——建议 A 之后再做 C。

候选 D · Board 与 export 共享一条 renderScene 通道

Candidate D

renderScene(ctx, scene) — 消掉重复渲染循环 + export 读私有色

Worth exploring
Files
src/board.tssrc/export.ts
Problem

export.ts 的 renderOffscreen 和 Board.render 的手写笔画循环重复：都是“填背景 → 遍历 strokes → drawStroke(ink 色)”。为了离屏渲染，export 直接读 board._bgColor / board._inkColor 这两个私有色字段。重复 + 伸手，两个味道叠一起。

Solution

抽出 renderScene(ctx, { strokes, viewport, bgColor, inkColor })——把 Board.render 和 export 都跑的那段收进一个纯函数。export 通过 Board 的 snapshot() 公共接缝拿 { strokes, colors }，不再读私有字段。Board.render 也调 renderScene。

BEFORE · 重复 + 读私有
Board.render循环 #1
export.renderOffscreen循环 #2
export→ board._bgColor/_inkColor →
同一段渲染逻辑两份；改一处易漏另一处。
→
AFTER · 一条通道 + 快照
renderScene(ctx,scene)
Board→ renderScene
export→ board.snapshot() → renderScene
渲染逻辑单源；私有色字段收回。
Benefits
  • Locality：笔画渲染规则单源；屏上/导出永远一致（WYSIWYG 更稳）。
  • 消灭泄漏：_bgColor / _inkColor 收回 private，export 经 snapshot() 接缝拿数据。
  • Tests：renderScene 是纯函数（ctx 可 mock），可断言绘制调用序列。
Deletion test · 中等。主要消灭重复 + 一处伸手。与 A 有协同：A 落地后 scene.strokes 直接来自 StrokeStore 快照，D 更自然。

Top recommendation

▶ 先做 Candidate A — StrokeStore

它是根，另外三个是枝

四个泄漏里，board.strokes 是唯一被三个模块穿透的，也是唯一让一整片纯几何逻辑无法 headless 单测的。A 落地后：C 的 apply/revert 能干净地落在 StrokeStore 上（C 变顺）；D 的 scene.strokes 直接取自 store 快照（D 变自然）。B 独立、便宜、可随时单独做——如果想先拿一个“今天就清一个红旗”的小胜，B 是最低成本的起点，但它不解决根。

建议顺序：A → D → C，B 任意时机穿插。四个做完，报告开头那句“只能全 public”就被逐条兑现成 private：A 封 strokes、B 封 render、C 把 _pushUndo 变公共 history.push、D 封 _bgColor/_inkColor。
